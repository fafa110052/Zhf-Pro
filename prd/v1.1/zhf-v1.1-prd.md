## 2. 业务背景与目标

### 2.1 业务背景

住好房展示平台 V1.0 已实现装修作品的展示与管理，但存在以下业务断层：

- **选材环节缺失**：业主浏览作品后无法直接选择硬装材料，需线下沟通，效率低下
- **人员分配无系统支撑**：设计师和监理的分配依赖人工协调，无系统化记录
- **订单追踪空白**：从选材申请到人员跟进无统一订单号追踪，容易遗漏

### 2.2 用户痛点

| 用户 | 痛点 |
|------|------|
| 业主 | 看中装修效果但不知道具体用什么材料；提交意向后长时间无反馈 |
| 设计师 | 不清楚被分配了哪些业主的选材需求 |
| 监理 | 无法系统查看自己的跟进任务 |
| 管理员 | 选材申请靠 Excel 管理，分配人员靠微信群，效率低易出错 |

### 2.3 业务目标

| 指标 | 当前值 | 目标值 |
|------|--------|--------|
| 选材申请处理效率 | 人工（无系统） | ≤2 个工作日完成审核分配 |
| 订单追踪覆盖率 | 0% | 100%（每单有订单号+操作日志） |
| 材料浏览到申请转化率 | — | ≥15% |

## 5. 详细功能需求

### 5.1 C 端微信小程序

#### 5.1.1 新增底部导航 Tab："在线选材"

**功能描述**
在现有"首页"、"分类"之后新增第 3 个 Tab"在线选材"，原第 3 个 Tab"我的"调整至第 4 位。图标使用材料/装修相关 icon。

**页面原型**
```
┌──────────────────────────────────┐
│  ████  ████  ████  ████         │
│  首页  分类  选材  我的            │
└──────────────────────────────────┘
```

**入口权限**
- 所有用户可访问，无需登录即可浏览楼盘和材料
- 未登录用户点击"提交申请"时，自动触发手机号一键登录流程

**交互逻辑**
1. 用户点击"在线选材"Tab → 进入楼盘筛选页面
2. Tab 图标使用 `selectedColor: #3b82f6`（与现有 Tab 一致）
3. 未登录用户在提交环节触发 `wx.login()` + `getPhoneNumber()` 快速登录

#### 5.1.2 楼盘筛选页面

**功能描述**
用户进入选材 Tab 后首先看到已开通选材功能的楼盘列表，可搜索和选择楼盘进入专属选材页面。

**页面原型**
```
┌──────────────────────────────────┐
│  🔍  搜索楼盘名称                  │  ← 搜索栏
├──────────────────────────────────┤
│  ┌──────────────────────────────┐│
│  │ 🏠  翡翠湖畔花园              ││
│  │     深圳市南山区科技园一路8号   ││  ← 楼盘卡片
│  │     [封面图]                  ││
│  └──────────────────────────────┘│
│  ┌──────────────────────────────┐│
│  │ 🏠  阳光海岸豪庭              ││
│  │     深圳市宝安区新安街道18号   ││
│  │     [封面图]                  ││
│  └──────────────────────────────┘│
│  ...                             │
└──────────────────────────────────┘
```

**交互逻辑**
1. 页面加载时调用 `GET /api/v1/properties?material_enabled=1` 获取已开通选材的楼盘列表
2. 每张楼盘卡片展示：封面图（`cover_image`）、楼盘名称、地址
3. 顶部搜索栏支持楼盘名称模糊搜索，输入 300ms 防抖后过滤列表
4. 点击楼盘卡片 → `navigateTo` 进入该楼盘的专属选材页面（传递 `property_id`）
5. 列表为空时展示空状态："暂无可用楼盘，敬请期待"

**异常处理**
- 接口超时/失败 → Toast "加载失败，请下拉重试"
- 搜索无结果 → 显示"未找到匹配的楼盘"

#### 5.1.3 楼盘专属选材页面

**功能描述**
用户选定楼盘后，进入该楼盘专属的材料选择页面。左侧分类导航固定悬浮，右侧展示对应分类的材料卡片。每类单选，底部悬浮栏显示已选汇总。

**页面原型**
```
┌──────────────────────────────────┐
│  翡翠湖畔花园 — 在线选材           │  ← 顶部楼盘名 + 搜索图标
│  🔍 搜索材料（点搜索图标展开）     │
├────────┬─────────────────────────┤
│ 地板   │ ┌─────────┐ ┌─────────┐ │
│▶墙面   │ │ 材料名   │ │ 材料名   │ │  ← 材料卡片（2列网格）
│ 天花板 │ │ 品牌A   │ │ 品牌B   │ │
│ 瓷砖   │ │ [图片]  │ │ [图片]  │ │
│ 卫浴   │ │ ¥128/㎡ │ │ ¥96/㎡  │ │
│ 橱柜   │ │  ✓已选  │ │         │ │  ← 选中态：蓝色边框+勾
│ 门窗   │ └─────────┘ └─────────┘ │
│        │ ┌─────────┐            │
│(可滚动)│ │ 材料名   │            │
│        │ │ 品牌C   │            │
│        │ │ [图片]  │            │
│        │ │ ¥156/㎡ │            │
│        │ └─────────┘            │
├────────┴─────────────────────────┤
│ 已选 3 类  ｜  预估总价 ¥352/㎡   │  ← 底部悬浮栏
│              [下一步 →]          │
└──────────────────────────────────┘
```

**交互逻辑**

*分类导航*
1. 左侧分类导航固定悬浮（`position: sticky`），上下可滚动
2. 分类列表从 `GET /api/v1/admin/material-categories` 获取，按 `sort_order` 排序
3. 点击分类 → 右侧区域滚动到对应分类锚点
4. 当前查看的分类高亮显示（蓝色左边框 + 加粗文字）

*材料展示*
1. 右侧材料列表按分类分组展示，每个分类标题下为两列网格
2. 材料卡片包含：材料名称、品牌、图片、单价（元/㎡或元/件）、简短描述
3. 支持页面级材料名称模糊搜索（点击顶部搜索图标展开搜索框）
4. 搜索结果保留分类结构——即搜索"地板"相关材料时，仍按分类分组展示

*选材逻辑*
1. **每类单选**：同一分类下只能选择 1 种材料
2. 选中后卡片显示蓝色边框 + 右上角蓝色勾选标记 ✓
3. 再次点击已选中材料 → 取消选中
4. 点击同分类其他材料 → 自动取消同分类旧选择，选中新材料
5. 选材数据存储在页面本地状态（不跨页面持久化）

*底部悬浮栏*
1. 固定于屏幕底部（`position: fixed; bottom: 0`）
2. 实时显示：已选材料分类数 + 已选材料单价总和
3. "下一步"按钮：至少选择 1 种材料后才可点击（未选择时灰色不可点）
4. 点击"下一步" → 跳转申请提交页面，传递已选材料数据（通过 `globalData` 或页面参数）

**异常处理**
- 某分类下无材料 → 显示"该分类暂无可用材料"
- 图片加载失败 → 显示占位图

#### 5.1.4 申请提交页面

**功能描述**
展示用户已选材料清单，填写房号、联系信息后提交申请，生成唯一订单号。

**页面原型**
```
┌──────────────────────────────────┐
│  提交选材申请                      │
├──────────────────────────────────┤
│  已选材料清单                      │
│  ┌──────────────────────────────┐│
│  │ 地板  │ 品牌A 实木复合地板     │ ¥128/㎡ │
│  │ 墙面  │ 品牌C 环保乳胶漆       │ ¥48/㎡  │
│  │ 卫浴  │ 品牌D 智能马桶套装     │ ¥3,280/件│
│  └──────────────────────────────┘│
│  预估总价：¥352/㎡ + ¥3,280/件     │
├──────────────────────────────────┤
│  房号 *                          │
│  ┌──────────────────────────────┐│
│  │ 如：3栋1单元1001              ││
│  └──────────────────────────────┘│
│                                  │
│  联系人姓名 *                     │
│  ┌──────────────────────────────┐│
│  │                              ││
│  └──────────────────────────────┘│
│                                  │
│  联系电话 *                       │
│  ┌──────────────────────────────┐│
│  │ 138****8888（自动填充可修改）  ││
│  └──────────────────────────────┘│
│                                  │
│  备注（选填，最多200字）           │
│  ┌──────────────────────────────┐│
│  │                              ││
│  └──────────────────────────────┘│
│                                  │
│  [ 提交申请 ]                    │
└──────────────────────────────────┘
```

**交互逻辑**

*表单预填*
1. 联系电话自动填充当前登录用户的手机号（从 `app.globalData.userInfo.phone` 获取），可手动修改
2. 联系人姓名自动填充当前用户昵称（如有），可手动修改

*订单号生成规则*
```
10位数字 = YYYYMMDD（6位日期）+ PP（2位小区编号）+ SS（2位当日自增序号）

示例：2026061201
  - 20260612 = 2026年6月12日
  - 01 = 小区编号（properties.property_code）
  - 01 = 当日该小区第1笔订单
```

订单号由后端生成，前端提交时无需拼接。后端逻辑：
1. 查询该小区当日最大订单号
2. 自增序号部分 +1（01-99，超出则提示当日已满）
3. 数据库唯一约束保证不重复

*提交校验*
1. 房号：必填，非空字符串，trim 后长度 ≥ 1
2. 联系人姓名：必填，非空字符串
3. 联系电话：必填，校验 `/^1[3-9]\d{9}$/`
4. 备注：选填，最多 200 字（`maxlength="200"`）
5. 未登录用户提交前自动触发登录

*提交成功页面*
```
┌──────────────────────────────────┐
│          ✓ 提交成功               │
│                                  │
│     订单号：202606120101          │
│     申请状态：待审核               │
│     预计 1-2 个工作日内回复         │
│                                  │
│    [ 查看我的申请 ]               │
│    [ 返回首页 ]                   │
└──────────────────────────────────┘
```

**异常处理**
- 订单号生成失败（序号超 99）→ Toast "今日申请已满，请明天再试"
- 表单校验失败 → 对应字段下方显示红色提示文字
- 网络错误 → Toast "提交失败，请重试"，保留已填表单数据

**审核流程**

*通过流程*
1. 点击"审核通过"→ 弹出分配人员窗口
2. 分配人员窗口：
   - 设计师下拉选择（从 `personnel_type=designer` 的人员列表获取）
   - 监理下拉选择（从 `personnel_type=supervisor` 的人员列表获取）
   - 两个下拉均支持搜索
   - 确认按钮
3. 确认后 → 调用 `POST /api/v1/admin/material-orders/:order_no/approve`
4. 后端更新订单状态为 `approved`，记录 `designer_id`、`supervisor_id`、`reviewed_by`、`reviewed_at`
5. 前端 Toast "审核通过" + 列表刷新
6. **依赖项**：触发微信模板消息通知申请人（需微信公众号模板消息权限，详见 10.4）

*驳回流程*
1. 点击"驳回申请"→ 弹出驳回原因窗口
2. 驳回原因窗口：
   - 预定义原因下拉（可选）：材料信息有误 / 楼盘未开通选材 / 申请人信息不完整 / 其他
   - 自定义原因文本框（必填，最多 500 字）
   - 取消/确认按钮
3. 确认后 → 调用 `POST /api/v1/admin/material-orders/:order_no/reject`，body 携带 `{ reason }`
4. 后端更新订单状态为 `rejected`，记录 `reject_reason`、`reviewed_by`、`reviewed_at`
5. C 端申请详情页同步显示驳回原因

*订单状态变更*
- 已通过订单 → 管理员可手动变更为"已完成"（`PATCH /api/v1/admin/material-orders/:order_no/complete`）
- 状态变更记录到操作日志

## 9. 非功能需求

### 9.1 兼容性

| 端 | 要求 |
|----|------|
| 微信小程序 | 兼容微信最新 3 个版本（基础库 ≥ 3.0.0） |
| 管理后台 | 兼容 Chrome 最新版、Edge 最新版 |

### 9.2 性能

| 指标 | 目标 |
|------|:---:|
| 小程序页面首屏加载 | ≤ 2 秒 |
| 材料列表 API 响应 | ≤ 500ms |
| 选材申请列表 API 响应 | ≤ 1 秒 |
| 图片上传 | ≤ 5 秒/张（≤ 10MB） |

### 9.3 安全性

1. **接口鉴权**：所有管理端接口和用户端写操作接口均需 JWT 认证
2. **RBAC**：Admin 接口使用 `requireRole('admin')` 中间件
3. **数据隔离**：用户端"我的申请"只能查询本人数据
4. **输入校验**：所有表单输入在前后端均做校验
5. **手机号脱敏**：列表展示时中间 4 位显示为 `****`
6. **SQL 注入防护**：使用 Knex.js 参数化查询

### 9.4 扩展性

1. **材料分类动态管理**：分类通过后台页面增删改，无需修改代码
2. **计价单位可配置**：`price_unit` 字段支持扩展（如 `/m`、`/套` 等）
3. **订单状态可扩展**：使用字符串枚举（非数字），方便增加新状态
4. **操作日志结构化**：`detail` 字段为 JSON 格式，支持未来扩展日志维度

## 10. 风险评估与应对措施

### 10.1 技术风险

| 风险 | 概率 | 影响 | 应对措施 |
|------|:---:|:---:|------|
| SQLite 并发写入瓶颈（订单号生成时） | 低 | 中 | 使用数据库事务 + 唯一约束 + 重试机制（最多 3 次重试） |
| 小程序第 4 个 Tab 导致底部导航过密 | 中 | 低 | UI 设计时控制 Tab 图标和文字大小，真机预览验证 |
| 图片量增多导致上传/加载变慢 | 中 | 中 | 复用现有 sharp 缩略图生成 + 小程序端图片压缩 |

### 10.2 业务风险

| 风险 | 概率 | 影响 | 应对措施 |
|------|:---:|:---:|------|
| 材料分类体系不适合实际业务 | 中 | 高 | 分类完全可配置，管理员可随时调整 |
| 订单号生成逻辑在高并发下出现重复 | 低 | 高 | 数据库 UNIQUE 约束 + 重试机制保障 |

### 10.3 时间风险

| 风险 | 概率 | 影响 | 应对措施 |
|------|:---:|:---:|------|
| 30 天开发周期不足 | 中 | 高 | Phase 可并行（如材料管理后端和楼盘管理后端可同步开发） |
| 小程序审核周期不可控 | 低 | 低 | V1.0 已有审核经验；本期仅迭代，审核较快 |

### 10.4 依赖项风险

| 风险 | 概率 | 影响 | 应对措施 |
|------|:---:|:---:|------|
| **微信模板消息权限未开通** | 高 | 中 | 标记为依赖项。审核通过后的通知降级为小程序内消息（在"我的申请"详情中显示审核结果）。后续申请到权限后再上线模板消息 |
