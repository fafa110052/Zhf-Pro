## 1. 产品概述

### 1.1 产品定位

住好房装修展示平台是一套面向装修公司的**双端作品展示系统**，由 C 端微信小程序（访客浏览 + 设计师创作）和 B 端 ERP 风格管理后台（管理员运营）组成。

### 1.2 目标用户

| 用户群 | 描述 | 使用端 | 登录方式 |
|--------|------|--------|----------|
| 普通访客 | 有装修需求的业主，浏览案例寻找灵感、评估装修公司能力 | C 端小程序 | 无需登录 |
| 公司设计师 | 装修公司设计师，微信一键授权登录后管理个人作品相册 | C 端小程序 | 微信一键授权 |
| 系统管理员 | 装修公司运营人员，审核作品、管理设计师权限、管理图片资产 | B 端网页后台 | 账号密码登录 |

### 1.3 MVP 版本范围

#### ✅ 包含功能

| 端 | 功能 |
|----|------|
| **小程序 — 访客** | 轮播图、热门推荐、按户型/部位/风格筛选、作品详情（高清图片轮播 + 基本信息） |
| **小程序 — 设计师** | 微信一键授权登录、个人中心、作品管理（上传/编辑/删除）、作品审核状态查看 |
| **管理后台** | ERP 风格界面、用户管理（设计师账号管理）、作品审核、图片库管理、基础数据统计 |

#### ❌ MVP 不包含

- 普通用户注册/登录（访客无需登录）
- 在线支付
- 评论、私信、收藏功能
- 在线咨询（预留接口，v2.0 实现）
- 预约量房（预留接口，v2.0 实现）
- 报价系统（预留接口，v2.0 实现）
- 复杂的数据分析报表（MVP 仅柱状图 + 饼状图）

### 1.4 强制约束

1. **零成本启动** — 前期完全免费部署和运行，SQLite 免数据库服务费，单机可运行
2. **极致可扩展性** — 模块化架构，未来可无缝迁移到付费服务器和备案域名
3. **纯 Node.js 技术栈** — 前后端分离，不使用任何其他后端语言
4. **SQLite 文件数据库** — 零配置，方便未来通过 Knex 迁移到 MySQL/PostgreSQL
5. **ERP 设计规范** — 管理端严格遵循：左侧垂直导航 + 顶部操作栏 + 数据表格布局
6. **预留接口** — 为在线咨询、预约量房、报价系统预留 API 扩展点

### 1.5 设计原则

1. **零成本优先** — 所有技术选型以免费/开源为第一原则
2. **访客零摩擦** — 浏览无需登录，打开即用
3. **设计师移动优先** — 案例管理完整集成于小程序，适配工地现场即时上传
4. **内容可控** — 设计师上传作品须经管理员审核后公开发布
5. **ERP 效率至上** — 管理后台严格遵循 ERP 交互范式
6. **迁移友好** — 所有数据层操作通过 Knex Query Builder，切换数据库仅改配置

## 3. 功能需求

### 3.1 C 端微信小程序

#### 3.1.1 首页（访客）

- **轮播图**：顶部全宽轮播 Banner，展示精选作品或公司宣传图（由管理员在后台配置），自动轮播 + 手动滑动
- **热门作品推荐**：展示管理员标记为"热门推荐"的作品，瀑布流布局，每项含封面图 + 标题 + 户型/风格标签
- **分类快捷入口**：三大筛选维度以图标卡片形式展示 — 户型 / 装修部位 / 风格

#### 3.1.2 分类筛选

- **按户型筛选**：一室 / 两室 / 三室 / 四室及以上 / 复式跃层 / 别墅
- **按装修部位筛选**：地板 / 墙壁 / 厨房 / 卫生间 / 阳台 / 客厅 / 卧室 / 书房 / 玄关 / 餐厅 / 全屋
- **按风格筛选**：现代简约 / 北欧 / 新中式 / 日式 / 美式 / 工业风 / 轻奢 / 地中海 / 混搭
- **交互方式**：顶部 Tab 切换维度 → 二级标签云选择具体分类 → 交叉筛选（可同时按户型 + 部位 + 风格组合）
- **结果列表**：两列瀑布流，每项展示封面图、作品名、设计师名、面积、造价区间

#### 3.1.3 作品详情

- **高清照片轮播**：全屏沉浸式滑动浏览实景照片，支持双指放大查看细节
- **基本信息展示**：作品名称、户型、面积（㎡）、装修部位、风格、造价区间（万元）、完工时间
- **设计师名片**：设计师头像、姓名、从业年限、个人简介
- **详情描述**：装修亮点、材料说明、设计理念（文本）

#### 3.1.4 设计师中心

**微信一键授权登录**
- 调用 `wx.getUserProfile()` 或微信手机号快速验证组件获取授权
- 首次登录：输入手机号与管理端预置的设计师信息校验 → 绑定 openid
- 后续登录：静默自动登录，无需重复操作

**个人信息**
- 展示/编辑：头像、姓名、手机号、从业年限、个人简介

**作品管理**
- **作品列表**：展示当前设计师全部作品，含状态标签 — 草稿 / 待审核 / 已发布 / 已驳回
- **上传新作品**：
  - 基本信息：作品名称、户型（下拉）、装修部位（下拉）、风格（下拉）、面积、造价区间、完工时间
  - 图片上传：手机相册选择或现场拍照，最多 20 张，拖拽排序，自动压缩
  - 描述文本：装修亮点、材料说明、设计理念
  - 提交后状态变为"待审核"；也可存为草稿稍后编辑
- **编辑作品**：修改信息或图片后重新进入"待审核"状态
- **删除作品**：弹窗二次确认后删除
- **审核状态跟踪**：实时查看作品当前审核进度

**个人数据统计**
- 作品总数、已发布数、待审核数、总浏览量

### 3.2 B 端网页管理后台（仅管理员）

> **设计总则**：管理端严格遵循 ERP 系统设计规范，参考 `erp-reference.png`。三大核心布局要素：**左侧垂直导航菜单** + **顶部操作栏** + **数据表格**。

#### 3.2.1 登录

- 账号密码登录，JWT Token 认证
- 登录态 7 天有效期
- 错误次数限制（5 次锁定 30 分钟）
- 仅 `role=admin` 账号可登录网页后台

#### 3.2.2 全局布局

```
┌──────────┬──────────────────────────────────────────────┐
│          │  [搜索框...] [查询] [重置]  [新增] [删除] [审核] [导出] │  ← 顶部操作栏
│  左侧    ├──────────────────────────────────────────────┤
│  垂直    │  ☑  │ 标题    │ 户型  │ 设计师 │ 状态  │ 操作     │
│  导航    │  ☑  │ 案例A   │ 三室  │ 张三   │ 待审核 │ 查看 编辑 删除 │  ← 数据表格
│  菜单    │  ☑  │ 案例B   │ 两室  │ 李四   │ 已发布 │ 查看 编辑 删除 │
│          │  ☐  │ ...    │ ...  │ ...   │ ...   │ ...        │
│          ├──────────────────────────────────────────────┤
│          │  < 1  2  3 ... 10 >  共 200 条               │  ← 分页器
└──────────┴──────────────────────────────────────────────┘
```

#### 3.2.3 左侧垂直导航菜单

| 菜单项 | 图标 | 功能说明 |
|--------|------|----------|
| **首页** | 🏠 | 仪表盘：概览卡片 + 趋势柱状图 + 分类饼状图 |
| **用户管理** | 👥 | 设计师账号管理（添加/删除/禁用/分配权限） |
| **作品管理** | 📋 | 全部作品列表 + 审核操作 + 批量管理 |
| **图片库管理** | 🖼️ | 全局图片资产统一管理 |
| **系统设置** | ⚙️ | 首页轮播图配置、热门推荐配置、系统参数 |

#### 3.2.4 顶部操作栏

每个管理页面顶部统一放置操作栏，包含：

- **搜索框**：文本输入，支持作品名称 / 设计师姓名模糊搜索
- **查询按钮**：执行筛选
- **重置按钮**：清空所有筛选条件
- 页面级操作按钮（按页面不同）：
  - 用户管理页：`[新增设计师]`
  - 作品管理页：`[审核通过] [审核驳回] [批量删除] [导出Excel]`
  - 图片库管理页：`[批量删除]`

#### 3.2.5 数据表格规范

所有列表页统一使用数据表格组件，包含以下要素：

- **复选框列**：首列，支持全选/反选
- **可排序列**：点击列头切换升序/降序
- **操作列**：末列，固定宽度，包含 `查看` `编辑` `删除` 等按钮
- **分页器**：底部居中，显示页码 + 上一页/下一页 + 总条数，默认 20 条/页，可选 50/100
- **行高亮**：鼠标悬停行背景变色
- **空状态**：无数据时显示插画 + "暂无数据"

#### 3.2.6 首页（仪表盘）

- **概览卡片**（4 个）：作品总数、本月新增、设计师人数、待审核数
- **柱状图**（Chart.js）：近 30 天作品发布趋势
- **饼状图**（Chart.js）：作品按风格分布占比
- **待审核提醒**：醒目展示当前待审核作品数，点击跳转作品管理

#### 3.2.7 用户管理

- **设计师列表**：表格含复选框、头像、姓名、手机号、作品数、状态（启用/禁用）、创建时间、操作列
- **添加设计师**：弹窗表单 — 姓名、手机号（必填，用于微信绑定校验）、从业年限
- **编辑设计师**：修改基本信息、重置微信绑定
- **禁用/启用**：禁用后该设计师所有作品自动下架
- **删除设计师**：二次确认；若有关联作品则提示并禁止删除

#### 3.2.8 作品管理

- **全部作品列表**：表格含复选框、封面缩略图、作品名称、户型、部位、风格、设计师、审核状态、发布时间、操作列
- **审核操作**：
  - 行内操作：每行操作列的"审核"按钮 → 弹出审核面板（左侧作品预览 + 右侧审核操作）
  - 审核通过：状态变更为"已发布"
  - 审核驳回：弹出驳回原因输入框（必填），状态变更为"已驳回"
  - 批量审核：勾选多行后点击顶部"审核通过"
- **作品详情查看**：点击行或"查看"按钮，展示完整作品信息 + 全部图片
- **强制下架**：将已发布作品改为已下架
- **批量删除**：勾选后批量删除，二次确认
- **导出 Excel**：将当前筛选结果导出为 Excel 文件
- **筛选条件**：顶部搜索框 + 审核状态下拉（全部/待审核/已发布/已驳回/已下架）+ 分类下拉 + 日期范围

#### 3.2.9 图片库管理

- **网格/列表双视图**：缩略图网格浏览 + 列表详情切换
- **筛选**：按设计师、上传时间范围、是否被引用筛选
- **图片详情**：查看大图、原始文件名、文件大小、上传者、被引用次数
- **删除**：仅允许删除"未被任何作品引用"的图片；已引用图片先提示
- **存储统计**：图片总数、占用磁盘空间

#### 3.2.10 系统设置

- **轮播图配置**：设置首页轮播图片（最多 5 张）+ 跳转链接
- **热门推荐配置**：勾选作品标记为"热门推荐"（拖拽排序，最多 10 个）
- **数据导出**：作品数据导出为 Excel

### 4.2 技术选型

| 层级 | 技术 | 版本 | 选型理由 |
|------|------|------|----------|
| **后端框架** | Express.js | 4.x | 轻量、生态成熟、学习成本低 |
| **API 风格** | RESTful JSON | — | 微信小程序原生支持，无需 GraphQL |
| **数据库** | SQLite (better-sqlite3) | 9.x | 零部署、零成本、同步 API 性能优异 |
| **ORM** | Knex.js (Query Builder) | 3.x | 比 ORM 更灵活，支持迁移，SQLite 兼容好 |
| **认证** | JWT (jsonwebtoken) | 9.x | 无状态，适合 API 服务 |
| **文件上传** | multer | 2.x | Express 生态标准，处理 multipart |
| **图片处理** | sharp | 0.33.x | 高性能缩略图生成 |
| **管理后台前端** | React 18 + React Router 6 | — | 组件化，生态丰富 |
| **CSS 框架** | TailwindCSS | 4.x | 原子化 CSS，适合 ERP 表格密集型 UI |
| **UI 组件** | 自建 ERP 组件 + Headless UI | — | 最小依赖，匹配 ERP 风格 |
| **图表** | Chart.js + react-chartjs-2 | — | 轻量、免费 |
| **富文本** | Tiptap | 2.x | 基于 ProseMirror，轻量可扩展 |
| **Excel 导出** | exceljs | 4.x | 纯 JS，无需依赖 Office |
| **C 端小程序** | 微信原生框架 | — | 无额外框架依赖，体积最小 |

## 5. 数据库设计

### 5.1 ER 关系

```
┌──────────────┐       ┌──────────────┐       ┌──────────────┐
│  designers   │       │    cases     │       │  categories  │
│  (设计师/管理员)│       │   (作品)      │       │  (分类字典)   │
└──────┬───────┘       └──────┬───────┘       └──────────────┘
       │                      │                      ▲
       │ 1:N                  │ N:1 (×3)             │
       ▼                      ▼                      │
┌──────────────┐       ┌──────────────┐              │
│    cases     │       │ case_images  │──────────────┘
│              │       │  (作品图片)    │   N:1 (×3)
└──────────────┘       └──────┬───────┘
                              │
                              │ N:1 (可选)
                              ▼
                       ┌──────────────┐
                       │ image_library│
                       │ (全局图片库)  │
                       └──────────────┘
```

> **说明**：`cases` 表通过 `house_type_id`、`area_category_id`、`style_category_id` 三个 FK 同时关联 `categories` 表。每条 `case_images` 记录可选关联 `image_library` 中的图片（通过 `library_image_id`）。

### 5.2 表结构定义

#### 5.2.1 `designers` — 设计师与管理员账号（统一用户表）

| 字段 | 类型 | 约束 | 说明 |
|------|------|------|------|
| `id` | INTEGER | PK, AUTOINCREMENT | 主键 |
| `openid` | VARCHAR(64) | UNIQUE, NULLABLE | 微信 openid（设计师登录凭证；管理员为 NULL） |
| `username` | VARCHAR(32) | UNIQUE, NULLABLE | 网页端登录账号（仅管理员） |
| `password_hash` | VARCHAR(128) | NULLABLE | bcrypt 密码哈希（仅管理员） |
| `name` | VARCHAR(32) | NOT NULL | 姓名 |
| `avatar_url` | VARCHAR(512) | — | 头像 |
| `phone` | VARCHAR(20) | UNIQUE, NOT NULL | 手机号（设计师微信绑定校验依据） |
| `years_of_exp` | INTEGER | DEFAULT 0 | 从业年限 |
| `bio` | TEXT | — | 个人简介 |
| `role` | VARCHAR(16) | DEFAULT 'designer' | 角色：designer / admin |
| `status` | VARCHAR(8) | DEFAULT 'active' | active / inactive |
| `is_bound` | INTEGER | DEFAULT 0 | 微信绑定状态 (0/1) |
| `login_attempts` | INTEGER | DEFAULT 0 | 登录错误次数 |
| `locked_until` | DATETIME | — | 锁定到期时间 |
| `created_at` | DATETIME | DEFAULT CURRENT_TIMESTAMP | — |
| `updated_at` | DATETIME | DEFAULT CURRENT_TIMESTAMP | — |

> **双通道登录**：同表承载两种角色。设计师（`role='designer'`）通过 openid 微信登录；管理员（`role='admin'`）通过 username + password 网页登录。管理员账号通过数据库 Seed 初始化。

#### 5.2.2 `categories` — 分类字典

| 字段 | 类型 | 约束 | 说明 |
|------|------|------|------|
| `id` | INTEGER | PK, AUTOINCREMENT | 主键 |
| `type` | VARCHAR(16) | NOT NULL | 分类维度：house_type / area / style |
| `name` | VARCHAR(32) | NOT NULL | 分类名称 |
| `sort_order` | INTEGER | DEFAULT 0 | 排序权重 |
| `is_active` | INTEGER | DEFAULT 1 | 启用状态 (0/1) |
| `created_at` | DATETIME | DEFAULT CURRENT_TIMESTAMP | — |

> **预置种子数据**：
> - `house_type`: 一室、两室、三室、四室及以上、复式跃层、别墅
> - `area`: 地板、墙壁、厨房、卫生间、阳台、客厅、卧室、书房、玄关、餐厅、全屋
> - `style`: 现代简约、北欧、新中式、日式、美式、工业风、轻奢、地中海、混搭

#### 5.2.3 `cases` — 装修作品

| 字段 | 类型 | 约束 | 说明 |
|------|------|------|------|
| `id` | INTEGER | PK, AUTOINCREMENT | 主键 |
| `title` | VARCHAR(128) | NOT NULL | 作品名称 |
| `description` | TEXT | — | 详情描述 |
| `house_type_id` | INTEGER | FK → categories.id | 户型 |
| `area_category_id` | INTEGER | FK → categories.id | 装修部位 |
| `style_category_id` | INTEGER | FK → categories.id | 风格 |
| `area_sqm` | DECIMAL(8,2) | — | 面积（㎡） |
| `budget_min` | INTEGER | — | 造价下限（万元） |
| `budget_max` | INTEGER | — | 造价上限（万元） |
| `completion_date` | DATE | — | 完工时间 |
| `designer_id` | INTEGER | FK → designers.id, NOT NULL | 所属设计师 |
| `cover_image` | VARCHAR(512) | — | 封面图路径 |
| `review_status` | VARCHAR(16) | DEFAULT 'draft' | 审核状态：draft / pending / approved / rejected / archived |
| `reject_reason` | TEXT | — | 驳回原因 |
| `reviewed_by` | INTEGER | FK → designers.id | 审核人 |
| `reviewed_at` | DATETIME | — | 审核时间 |
| `is_hot` | INTEGER | DEFAULT 0 | 是否热门推荐 |
| `view_count` | INTEGER | DEFAULT 0 | 浏览次数 |
| `created_at` | DATETIME | DEFAULT CURRENT_TIMESTAMP | — |
| `updated_at` | DATETIME | DEFAULT CURRENT_TIMESTAMP | — |

**索引**：
- `idx_cases_review_status` ON (`review_status`)
- `idx_cases_house_type` ON (`house_type_id`)
- `idx_cases_area` ON (`area_category_id`)
- `idx_cases_style` ON (`style_category_id`)
- `idx_cases_designer` ON (`designer_id`)
- `idx_cases_hot` ON (`is_hot`, `review_status`)
- `idx_cases_created` ON (`created_at`)

**审核状态流转**：
```
draft → pending → approved ⇄ archived
            ↘ rejected → draft (修改后)
```

#### 5.2.4 `case_images` — 作品图片

| 字段 | 类型 | 约束 | 说明 |
|------|------|------|------|
| `id` | INTEGER | PK, AUTOINCREMENT | 主键 |
| `case_id` | INTEGER | FK → cases.id, ON DELETE CASCADE | 所属作品 |
| `library_image_id` | INTEGER | FK → image_library.id, NULLABLE | 关联图片库 |
| `image_url` | VARCHAR(512) | NOT NULL | 原图路径 |
| `thumb_url` | VARCHAR(512) | — | 缩略图路径 |
| `sort_order` | INTEGER | DEFAULT 0 | 排序 |
| `created_at` | DATETIME | DEFAULT CURRENT_TIMESTAMP | — |

#### 5.2.5 `image_library` — 全局图片库

| 字段 | 类型 | 约束 | 说明 |

### 5.3 数据库迁移策略

当前使用 SQLite，通过 **Knex.js** 统一数据访问层。未来迁移到 MySQL/PostgreSQL 路径：

1. 所有表操作均通过 Knex Query Builder，不写原生 SQL
2. 迁移时仅需修改 `knexfile.js` 中的 `client` 和 `connection` 配置
3. 运行 `knex migrate:latest` 在新数据库重建表结构
4. 运行数据迁移脚本完成 SQLite → 新数据库数据同步

│  │        Nginx (反向代理 + 静态文件)        │ │
│  │  ┌──────────────────────────────────┐   │ │

## 8. 扩展性设计与预留接口

### 8.1 模块化架构

- 后端按功能模块拆分路由和服务层（`routes/` + `services/`），新功能只需新增模块文件并注册路由
- 前端管理后台采用页面级组件隔离，新增页面仅需三步：创建页面组件 → 注册路由 → 添加导航菜单项
- 小程序页面按功能独立分包，避免主包体积膨胀

### 8.2 分类体系扩展

- `categories` 表的 `type` 字段采用字符串而非枚举，新增分类维度只需插入新 `type` 值
- 前端筛选组件根据 API 返回的 `type` 动态渲染分类入口
- v2.0 可引入 `case_category_map` 中间表实现 N:N 灵活关联

### 8.3 数据库迁移路径

通过 **Knex.js** 抽象数据访问层，从 SQLite 迁移到 MySQL/PostgreSQL 仅需：

```
1. 修改 knexfile.js: client: 'mysql2' / 'pg'
2. 修改 connection 配置（host/port/user/password/database）
3. knex migrate:latest → 在新数据库重建结构
4. 编写数据迁移脚本同步存量数据
```

无需修改任何业务代码。

### 8.4 服务器迁移路径

| 阶段 | 方案 | 月成本 | 适用场景 |
|------|------|:------:|----------|
| **MVP** | 本地开发机 + Cloudflare Tunnel | ¥0 | 开发测试 |
| **初期** | 腾讯云/阿里云 1C2G 轻量服务器 | ~¥30-50 | 小规模运营 |
| **增长** | 升级云服务器配置 + 自定义域名备案 | ~¥100-200 | 正式商用 |
| **规模** | 容器化部署 + 负载均衡 + OSS | 按需 | 大规模运营 |
