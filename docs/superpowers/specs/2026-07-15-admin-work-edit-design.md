# 设计文档:管理后台"编辑作品"功能

- 日期:2026-07-15
- 状态:已与用户确认方案A
- 关联:作品图片上限15张(2026-07-15 已上线)、创建弹窗孤儿图清理(2026-07-15 已上线)

## 背景与目标

管理后台作品管理目前只能创建、审核、上下架作品,不能修改已有作品的信息。需要新增"编辑作品"能力:可修改作品全部信息,图片可增删(总数不超过 15 张),复用创建弹窗已有的上传限流与图库防堆积机制。

## 已确认的业务决策

| 决策点 | 结论 |
|--------|------|
| 编辑已通过的作品后 | **直接生效**,不重新走审核(管理员为最高权限) |
| 可编辑字段 | 标题/说明/户型/空间/风格/面积/预算/图片/封面;**设计师归属不可改** |
| 编辑中删除的旧图 | 保存成功后**同步从图库删除**;若被其他地方引用(首页 banner、其他作品等)则自动跳过不删 |

## 方案(A):创建弹窗改造为创建/编辑两用

### 前端(admin/src/pages/Works.jsx)

**组件改造**:`CreateModal` 重命名为 `WorkFormModal`,新增 `work` 入参:
- `work == null` → 创建模式(现有行为完全不变)
- `work != null` → 编辑模式,打开时 `GET /admin/works/:id` 拉取详情预填表单与已有图片

**入口(两处)**:
1. 作品列表每行操作区新增"编辑"按钮
2. 详情面板(DetailPanel)新增"编辑"按钮

**编辑模式的图片状态模型**:
- `existingImages`:已有图片 `[{ id, image_url, thumb_url, removed: false }]`,点 X 仅置 `removed: true` 并从展示列表隐藏(与新图删除的交互一致),**不立即调删除接口**——防止改一半取消导致线上作品裂图
- `uploadedImages`:本次新增图片,复用现有逻辑(选中即上传、同时最多 3 张限流、进度条、失败标错)
- 上限校验:`未标记删除的旧图数 + 新图数 ≤ MAX_IMAGES(15)`,超出提示并截断
- 封面:旧图与新图统一可点选为封面;若被删图恰是封面,自动回退到剩余第一张

**取消(未保存关闭)**:
- 仅删除本次新上传的图(`uploadedImages` 中有 id 的逐张 `DELETE /admin/images/:id`),与创建模式同一套防堆积逻辑
- 已有旧图不做任何操作(含被标记删除的)
- 图片上传中禁止关闭弹窗(现有逻辑)

**保存**:
1. `PUT /admin/works/:id` 提交全部字段 + 最终图片 ID 列表(保留的旧图 + 新图,顺序即展示顺序)+ `cover_image`
2. 成功后:对被标记删除的旧图逐张 `DELETE /admin/images/:id`(不带 force;被引用时后端拒绝,前端静默忽略失败)
3. 刷新列表与详情面板,toast 提示"作品已更新"
4. 失败:弹窗保留全部内容,展示错误信息,可重试;不删除任何旧图

**编辑模式差异细节**:
- 设计师下拉:显示当前设计师但 disabled
- VR 链接**不进此弹窗**——详情面板已有独立的 VR 链接编辑保存(并行开发的功能),避免两处写同一字段

### 后端(server)

**新增路由** `PUT /api/v1/admin/works/:id`(routes/cases.js):
- `authenticate + requireRole('admin')`
- 从 body 中**剔除 `designer_id`**(接口层防误改归属)
- 复用 `caseService.update(id, data)`(小程序设计师编辑同款逻辑,已支持 images 整组替换 case_images)
- **不改变 `review_status`**:实施时核实 `caseService.update` 不会重置审核状态;若现有实现有状态副作用(如设计师编辑后回草稿),管理员路径需绕过该副作用
- 响应:`{ success: true, data: 更新后的作品 }`

无数据库迁移。

## 错误处理

| 场景 | 行为 |
|------|------|
| 详情加载失败 | 弹窗内显示错误 + 重试,不允许保存 |
| 保存失败(网络/校验) | 保留表单内容与图片状态,展示错误,可重试 |
| 删除旧图失败(被引用/网络) | 静默跳过,不影响保存成功的结果 |
| 新图部分上传失败 | 标错可单张移除,保存时只提交成功的图 |

## 验收标准

1. 编辑一个已通过的作品:改标题、新增 2 张图、删 1 张旧图、换封面 → 保存 → 小程序端立即看到变化,审核状态仍为"已通过"
2. 编辑改到一半点取消 → 原作品完好(信息、图片、封面均无变化),图库中无本次产生的无主图片
3. 已有 14 张图时只能再加 1 张,多选自动截断并准确提示
4. 被首页 banner 引用的图从作品中删除后,banner 不裂图(图库中该图保留)
5. 后台构建通过,部署生产后回归创建作品流程无回归

## 风险与注意

- **并行开发冲突**:VR 看房功能同期改动 Works.jsx 与后端 cases 路由,实施前先拉取最新 main,基于最新代码开发
- `caseService.update` 的审核状态副作用需在实施第一步核实(见后端一节)
- 部署走既有流程:构建 admin + pm2 restart(本功能含后端路由改动,需重启后端)
